# AI群聊发现器

> 内部 AI 应用官方群聊发现工具，用于帮助调研人员快速找到目标 AI 产品的官方群入口。

## 这个项目是干什么的

[引用：产品总纲 §1]

AI群聊发现器面向内部调研和信息获取场景，核心目标是减少人工逐个翻官网、翻 GitHub 找官方群入口的时间成本。

它解决的是一个非常具体的问题：
- 当你想找 `Cursor`、`Claude`、`Lovable` 这类 AI 产品的官方群时，不需要再手动翻多个站点。
- 系统会优先在官网首页、官网底部相关页面、GitHub 官方主页和 README 中查找官方群入口。
- 如果识别到的是“联系我们”、公众号二维码或非官方入口，系统不会把它们当作官方群结果返回。

适用人群：
- AI 产经调研人员
- 产品经理
- 行业研究员

## MVP 范围

[引用：产品总纲 §3，§4，§5，§12]

### 已支持

| 类型 | 内容 |
|------|------|
| 群平台 | 微信 / QQ / 飞书 |
| 搜索输入 | 产品名 / 英文别名或常见别名 / 官网域名 |
| 优先数据源 | 官网首页、官网 footer / contact / community 页面、GitHub 组织主页、GitHub 仓库 README |
| 结果形态 | 一个产品一个主卡片，可展开查看多个官方群 |
| 筛选与排序 | 群平台、群类型、隐藏已添加、热门、最新 |
| 更新方式 | 弱更新：每次搜索实时抓取 |

### 后续预留（当前未启用）

| 类型 | 内容 |
|------|------|
| 推荐关键词入口 | 未来可在首页或搜索框附近展示可点击推荐词，用于探索模式 |
| 默认行为 | 推荐词先扩展为候选产品，再复用现有官方群发现流程 |
| 边界 | 不直接把泛关键词当作官方群结果，不直接全网搜索二维码 |

### 暂不支持

| 类型 | 说明 |
|------|------|
| 人工补录 | 当前目标是自动化发现，不做手工补充入口 |
| 登录体系 | MVP 为本地单用户使用，无需登录 |
| 团队共享状态 | 只保存当前用户本地状态 |
| 推荐关键词生成 / 复杂推荐算法 | 当前不支持；排序阶段仅使用 GitHub stars |
| 钉钉 / Telegram / Discord | 不在当前 MVP 平台范围内 |
| 非官方群收录 | 不展示民间群、KOL 群、泛社区入口 |

## 怎么运行

[引用：ARCHITECTURE.md.md §8]

当前建议采用本地 Web 工具形态运行：前端使用 `React + Vite + TypeScript`，本地抓取服务使用 `FastAPI`。

### 环境要求

```text
Node.js >= 20
Python >= 3.11
Playwright Chromium
SQLite（本地文件，无需单独安装服务）
```

### 启动步骤

```bash
# 1. 进入前端目录
cd app/frontend

# 2. 安装前端依赖
npm install

# 3. 启动前端
npm run dev
```

```bash
# 1. 进入后端目录
cd app/backend

# 2. 创建并激活 Python 虚拟环境
python -m venv .venv
.venv\Scripts\activate

# 3. 安装后端依赖
pip install -r requirements.txt

# 4. 安装浏览器运行时
playwright install chromium

# 5. 启动本地抓取服务
uvicorn main:app --reload --port 8000
```

### 运行说明
- 前端负责搜索框、筛选栏、主卡片和二维码查看等界面能力。
- 本地抓取服务负责抓取官网 / GitHub 页面、提取二维码或官方群链接、写入本地状态。
- “已添加”状态与搜索历史默认保存在本地，不依赖远程服务。

## 怎么搜索

[引用：产品总纲 §5，§7，§8]

### 基本搜索

1. 在顶部搜索框输入产品名、英文别名或官网域名。
2. 按 Enter、点击搜索按钮，或在候选列表中选择某个候选产品后，正式提交搜索。
3. 提交搜索后，系统会优先抓取官网和 GitHub 官方页面。
4. 若结果存在产品名歧义，系统会先给出最多 3 个候选产品。
5. 识别成功后，结果区返回主卡片列表；点击展开可以查看同一产品下的多个官方群。

补充说明：
- 当前版本只支持产品名 / 别名 / 官网域名作为正式输入。
- 输入过程中可以出现候选产品提示，但不会在仅输入时直接触发正式抓取。
- “推荐关键词”是后续预留能力，不代表当前 README 所描述的已支持功能。

### 结果阅读方式
- 主卡片展示应用名称、应用简介、GitHub stars、创建时间、最近验证时间。
- 入群方式优先显示二维码缩略图；点击后可放大查看。
- 如果没有二维码但有官方群入口链接，会显示“二维码暂未抓取成功”，并保留入群入口。

### 筛选与排序

| 操作 | 结果 |
|------|------|
| 按平台筛选 | 在微信 / QQ / 飞书官方群之间筛选 |
| 按群类型筛选 | 在交流群 / 答疑群 / 售后群 / 招募/内测群之间筛选 |
| 点击“热门” | 按 GitHub stars 重排当前结果 |
| 点击“最新” | 按最近验证时间重排当前结果 |
| 保持“隐藏已添加”开启 | 默认不显示已添加结果 |
| 关闭“隐藏已添加” | 已添加结果重新出现，但会弱化显示 |

### 搜索失败
- 若系统未在官网或 GitHub 官方页面中识别到官方群入口，结果区直接显示“未发现官方群”。
- 系统不会用“联系我们”、公众号二维码或非官方社区入口代替官方群结果。

### 当前支持的来源

| 来源 | 当前状态 |
|------|----------|
| 官网首页 | 支持 |
| 官网 footer / contact / community 页面 | 支持 |
| GitHub 组织主页 | 支持 |
| GitHub 仓库 README | 支持 |
| 官方文档站 | 暂不纳入当前 MVP |

### 官方群识别口径
- 入口必须出现在官网或 GitHub 官方页面。
- 周边文案必须明确指向交流、社群、答疑、用户群、开发者群等官方群语义。
- 只有“联系我们”、只有公众号二维码、只有第三方社区链接的页面，不计入官方群结果。

## 已知限制

[引用：产品总纲 §6，§11，§12]

| 限制 | 说明 |
|------|------|
| 仅收录官方群 | 民间群、KOL 群、泛社区入口不在范围内 |
| 无登录态 | 本地单用户使用，换设备后状态不会自动同步 |
| 弱更新 | 每次搜索重新抓取，不依赖旧缓存直接展示结果 |
| 二维码可能变更 | 最近验证时间只能说明本次抓取时间，不能保证二维码长期有效 |
| 无公开仓库的产品 | GitHub stars 可能显示“—”或“无公开仓库” |
| 群入口识别依赖官方页面露出 | 如果官方没有公开露出官方群入口，系统不会补猜或补录 |
