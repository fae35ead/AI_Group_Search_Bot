# 1. 产品目标与边界

[引用：产品总纲 §1，§12]

AI群聊发现器是一款面向内部调研场景的 AI 产品官方群发现工具，目标是在尽量少人工介入的前提下，帮助用户快速定位并进入目标 AI 产品的官方群聊。

本产品当前阶段的核心价值有三点：
- 降低人工逐站点翻找官方群入口的时间成本。
- 用统一口径过滤掉“联系我们”、公众号二维码、民间社群等无效入口。
- 通过每次搜索实时抓取的方式，尽量减少展示过期二维码或失效群入口的概率。

MVP 的明确边界如下：
- 做：搜索具体产品的官方群入口，支持查看热门/最新结果，支持对结果进行平台、群类型与已添加状态筛选。
- 做：仅收录官网或 GitHub 官方页面中出现的官方群入口，优先展示二维码，没有二维码时展示入口链接。
- 做：为后续“推荐关键词”探索入口保留产品口径与文档扩展位，但不纳入当前 MVP 交付范围。
- 不做：人工补录、登录体系、团队共享状态、推荐关键词生成、复杂热门算法、复杂推荐算法、泛社区发现、非官方群收录。
- 不做：钉钉、Telegram、Discord 等非 MVP 平台。

# 2. 用户场景

[引用：产品总纲 §1]

本产品服务于 AI 产经调研人员、产品经理和行业研究员，主要覆盖以下三类场景：

### 场景 A：搜索具体产品名，找到官方群入口
- 用户输入产品名、英文别名或官网域名，例如 `Cursor`、`Claude`、`lovable.dev`。
- 系统优先从官网首页、官网底部相关页面、GitHub 官方主页与 README 中查找官方群线索。
- 用户预期结果是直接看到可进入的官方群二维码或官方群链接，而不是泛泛的社区入口。

### 场景 B：浏览热门或最新官方群结果
- 用户在已有搜索结果基础上，通过“热门”或“最新”排序快速浏览更值得关注的结果。
- “热门”优先用于发现行业影响力较大的 AI 产品官方群。
- “最新”优先用于发现最近一次被成功验证过的官方群入口。

### 场景 C：过滤已添加结果，减少重复查看
- 用户在看过某个群或已经加入某个群后，可将该群标记为“已添加”。
- 系统默认隐藏已添加结果，并提供查看已隐藏结果的开关。
- 隐藏粒度仅针对当前群，不影响同一产品下的其他官方群继续展示。

### 场景 D：通过推荐关键词进入探索模式（后续迭代，不纳入当前 MVP）
- 当用户暂时不知道具体产品名时，系统可在首页或搜索框附近展示一组推荐关键词。
- 用户点击推荐关键词后，系统先扩展出候选产品，再复用现有官方群发现流程。
- 推荐关键词的输出结果仍然是现有主卡片列表，不新增独立结果类型。
- 推荐关键词不应被定义为“直接全网搜二维码”，仍只收录官网或 GitHub 官方页面中的官方群入口。

# 3. 输入输出

[引用：产品总纲 §5]

### 3.1 搜索输入

| 输入形式 | 示例 | 说明 |
|----------|------|------|
| 产品名 | Cursor | 支持中文名或英文名 |
| 英文别名 / 常见别名 | Claude | 用于补充产品名歧义或品牌缩写 |
| 官网域名 | lovable.dev | 用于直接定位官方站点 |

当前 MVP 仅支持以上三类输入。

正式搜索触发规则：
- 用户按 Enter；
- 用户点击搜索按钮；
- 用户在候选列表中选中某个候选产品。

补充约束：
- 输入过程可用于更新候选产品提示；
- 输入过程本身不直接触发正式抓取。

### 3.1.1 后续扩展：推荐关键词入口（不纳入当前 MVP）
- 功能名称：`推荐关键词`
- 功能定位：探索型入口，帮助用户在不知道具体产品名时启动官方群发现。
- 推荐形态：首页或搜索框附近展示可点击主题词，而不是替代当前默认搜索输入。
- 点击行为：推荐词 → 候选产品 → 官方页面官方群发现。
- 初版示例：`热门 AI 编程助手`、`开源 AI Agent`、`AI 视频生成`、`最近新增 AI IDE`
- 边界要求：不直接把泛关键词当作官方群结果，不展示非官方页面发现的社群。

### 3.2 搜索输出

| 场景      | 输出形式   | 说明                     |
| ------- | ------ | ---------------------- |
| 未发现官方群  | 空结果态   | 明确提示“未发现官方群”           |
| 产品名存在歧义 | 候选产品列表 | 默认返回前 3 个候选，由用户手动选择    |
| 成功识别    | 主卡片列表  | 一个产品对应一个主卡片，可展开查看多个官方群 |

### 3.3 输出行为要求
- 结果卡片不向用户展示来源链接。
- 二维码优先展示；没有二维码但存在官方群入口时，展示入口链接并标注“二维码暂未抓取成功”。
- 搜索失败时不以“联系我们”、公众号二维码或非官方社区入口代替官方群结果。

# 4. 结果卡片字段

[引用：产品总纲 §6]

| 字段名          | 说明                            | 空值处理                     | 来源            |
| ------------ | ----------------------------- | ------------------------ | ------------- |
| 应用名称         | 当前结果所属的 AI 产品名称               | 无空值方案，按匹配结果返回            | 搜索词匹配结果       |
| 入群方式         | 群二维码或官方群入口链接，二维码优先            | 无二维码时展示链接，并标注“二维码暂未抓取成功” | 官方页面抓取结果      |
| 群类型          | 交流群 / 答疑群 / 售后群 / 招募/内测群 / 未知 | 无法判断时显示“未知”              | 周边文案识别        |
| 应用简介         | 官网或 GitHub 的一句官方描述            | 显示“—”                    | 官网或 GitHub 抓取 |
| GitHub stars | 官方主仓库 stars                   | 显示“—”或“无公开仓库”            | GitHub 官方仓库   |
| 创建时间         | 官方主仓库创建时间                     | 显示“未知”                   | GitHub 官方仓库   |
| 最近验证时间       | 本次搜索成功抓取到官方群入口的时间             | 无空值方案，按本次抓取时间记录          | 系统生成          |

### 4.1 字段使用规则
- 同一产品存在多个官方群时，主卡片承载产品级信息，展开后展示多个群入口。
- 字段命名与展示口径必须与产品总纲保持一致，不引入额外用户可见字段。
- “最近验证时间”用于表达本次结果的新鲜度，不等价于群二维码永久有效。

# 5. 识别规则

[引用：产品总纲 §4，§5]

### 5.1 官方群识别规则
- 识别范围限定为：官网首页、官网 footer / contact / community 页面、GitHub 组织主页、GitHub 仓库 README。
- 入口必须来自官网或 GitHub 官方页面，且周边文案明确指向交流、社群、答疑、用户群、开发者群等官方群语义。
- 当页面既存在群入口也存在公众号、联系邮箱等其他入口时，仅保留官方群入口用于展示。

### 5.2 不展示的内容
- 只有“联系我们”但没有明确官方群指向的页面。
- 只有公众号二维码但没有明确群聊语义的页面。
- 民间群、KOL 群、论坛、第三方社区聚合页。
- 无法证明属于官网或 GitHub 官方页面的入口。

### 5.3 群类型枚举

| 群类型 | 判定依据 |
|--------|----------|
| 交流群 | 周边文案出现“交流”“社区”“Community”“Join us”等表达 |
| 答疑群 | 周边文案出现“答疑”“Q&A”“Support”等表达 |
| 售后群 | 周边文案出现“售后”“客服”“Help”等表达 |
| 招募/内测群 | 周边文案出现“招募”“内测”“Beta”等表达 |
| 未知 | 没有足够线索归入以上任一类型 |

# 6. 筛选与排序

[引用：产品总纲 §8]

### 6.1 筛选项

| 类别 | 选项 |
|------|------|
| 群平台 | 微信 / QQ / 飞书 |
| 群类型 | 交流群 / 答疑群 / 售后群 / 招募/内测群 |
| 状态开关 | 隐藏已添加（开 / 关） |
| 排序按钮 | 热门 / 最新 |

### 6.2 排序规则
- 默认结果排序：
  1. 产品名匹配度
  2. 是否存在明确官方群二维码
  3. GitHub stars
- “热门”排序：在当前筛选结果内，优先按 GitHub stars 排序。
- “最新”排序：在当前筛选结果内，按最近验证时间倒序展示。

### 6.3 筛选逻辑要求
- 平台和群类型筛选只针对已识别出的官方群结果生效。
- “隐藏已添加”默认开启，关闭后可查看已隐藏结果。
- 单个群被标记为“已添加”后，不应连带隐藏同产品下的其他群。

# 7. 边界与异常处理

[引用：产品总纲 §5，§12]

| 场景 | 处理方式 |
|------|----------|
| 未发现官方群 | 显示“未发现官方群”，返回空结果 |
| 产品名歧义 | 返回前 3 个候选产品，由用户自行选择 |
| 无二维码但有官方群链接 | 展示入口链接，并标注“二维码暂未抓取成功” |
| GitHub stars 为空 | 显示“—”或“无公开仓库” |
| 创建时间为空 | 显示“未知” |
| 只有“联系我们”或公众号入口 | 不展示，不作为官方群结果返回 |
| 已添加结果 | 默认隐藏，并提供查看已隐藏结果的开关 |
| 当前群被隐藏 | 仅隐藏当前群，不影响同产品下其他群 |
| 每次重新搜索 | 实时抓取并刷新最近验证时间 |

### 7.1 异常处理原则
- 宁缺毋滥：宁可不返回结果，也不返回不确定的官方群入口。
- 不使用非官方群入口充当替代结果。
- 不向用户暴露技术性异常细节，只展示与任务相关的结果态和空结果态。

# 8. MVP 不做清单

[引用：产品总纲 §12]

以下内容不纳入当前 MVP 范围：
- [ ] 人工补录入口
- [ ] 用户登录、注册、权限系统
- [ ] 团队共享状态与多用户协作
- [ ] 推荐关键词生成
- [ ] 复杂热门算法或复杂推荐算法（后续若做推荐关键词，初版也应优先规则化实现）
- [ ] 钉钉 / Telegram / Discord 平台支持
- [ ] 泛社区发现与第三方社区收录
- [ ] 非官方群、民间群、KOL 群展示
- [ ] 面向用户展示来源链接
