## 问题
###### 官方群聊的定义
只接受官方运营的群，如各网站、Github主页最底下的群聊二维码。

###### 覆盖平台优先级
- 优先做：微信群、QQ群、飞书群
- 后做：钉钉群、Telegram、Discord
微信群是核心。

###### 使用入口
优先级排序：
- 先做：搜一个具体产品，比如：“我想找 Cursor / Manus / Lovable 的官方群
- 后做：看最新热门，比如：“给我最近一周新增、热度高的 AI 应用官方群”，但该功能实现用按钮筛选的方式实现，而非搜索交互
- 再后做：直接生成推荐关键词。

补充说明：
- “推荐关键词”是**后续迭代能力**，不是当前默认搜索输入。
- 它的定位是探索型入口：在用户不知道具体产品名时，先给出一组可点击的推荐词。
- 用户点击推荐词后，系统先扩展出候选产品，再复用现有官方群发现流程。
- 它不等于“直接全网搜二维码”，仍然只认官网 / GitHub 官方页面中的官方群入口。

###### 结果卡片
- 必须字段：应用名称、入群方式（以群聊二维码为主）、群类型、应用简介、Github星级、创建时间、最近验证时间

###### 已添加按钮的业务含义
我已经看过了，不想再看 / 我已经加入这个群了
暂时无需拆开成两个状态，目的只是为了防止每次搜索都推荐相同的群聊。

###### 卡片信息更新
弱更新：用户每次搜索时，临时去抓一次最新网页信息

###### 应用的Github星级
如果有官方主仓库，则显示主仓库 stars；创建时间默认显示仓库创建时间

###### ”热门“的定义
量化指标：
- GitHub stars
- 其他后续再开发

###### “推荐关键词”的定义
- 功能定位：面向探索场景的后续能力，用于自动生成可点击的主题词。
- 与“热门 / 最新”的区别：
  - “热门 / 最新”是当前结果内的排序能力。
  - “推荐关键词”是未来的探索入口能力。
- 当前阶段：不作为默认搜索入口，不纳入 MVP 正式需求范围。
- 后续交互：在首页或搜索框附近展示推荐词，点击后进入“探索模式”。
- 后续默认实现路径：推荐词 → 候选产品 → 官方页面官方群发现。
- 初版实现不要求复杂推荐算法，可先基于规则、预置词库和热门源生成。
- 示例：`热门 AI 编程助手`、`开源 AI Agent`、`AI 视频生成`、`最近新增 AI IDE`

###### 搜索失败处理
明确提示”未发现官方群“，返回空即可。

###### 人工补录
我认为不需要。因为我们就是想要自动化搜索群聊，减少人工搜索的成本。

###### 搜索范围
- 优先搜索：**官网首页**、官网 footer / contact / community 页面、GitHub 组织主页、GitHub 仓库 README
- 后搜索：官方文档站、官方招聘页、官方公众号文章页

###### 群聊识别
中等严格：只要二维码出现在官网/GitHub官方页面，且周边文字明显表示“交流、社群、答疑、用户群、开发者群”，就算。

###### 群类型
- 交流群
- 答疑群
- 售后群
- 招募/内测群
- 未知

###### 展示页面
- 同时展示二维码缩略图 + 放大查看
- 二维码优先，但没有二维码时可展示官方群入口链接
- 不展示来源链接
- 返回结果为一个产品一个主卡片，展开后看多个群

###### 隐藏方式
只隐藏当前这个群卡片，其他群仍可显示

###### 搜索输入支持形式
目前：
- 产品名
- 英文别名 / 常见别名
- 官网域名

后续扩展：
- 推荐关键词入口（例如：`热门 AI 编程助手`、`开源 AI Agent`）
- 该入口属于未来能力预留，不改变当前“产品名 / 别名 / 域名”的搜索定义

###### 应用简介来源
优先用官网或 GitHub 的一句官方描述

###### 创建时间为空的处理方式
显示未知

###### Github Star为空的处理方式
显示 “—”或“无公开仓库”。

###### 最近验证时间
系统本次搜索成功抓到该官方群入口的时间，该指标的目的是为了防止群本身二维码被替换，但返回的结果仍然是旧的二维码。

###### ”热门“/”最新“按钮位置
产品形态希望是，最顶部为搜索框，搜索框下方为筛选栏。在搜索后，筛选栏下方出现返回结果。故这两个按钮位置应该放在搜索框下方。

###### 看过的处理逻辑
默认隐藏已添加，并提供“查看已隐藏结果”的开关。

###### 搜索结果数量上限
默认返回前3个产品候选

###### 产品名歧义
返回多个候选让用户自己选

###### 没有官方群，但有官方“联系我们/公众号/社群页”时怎么处理
不展示

###### 二维码抓取失败的处理逻辑
展示入口链接，并标注“二维码暂未抓取成功”

###### ”已添加“的保存策略
当前用户长期保存

###### 登陆状态
无需登录，直接本地使用抓取信息

###### 普通搜索结果排序
- 产品名匹配度
- 是否有明确官方群二维码
- GitHub stars

###### 筛选项
- 群平台：微信 / QQ / 飞书
- 群类型：交流群 / 答疑群 / 售后群 / 招募/内测群
- 隐藏已添加开关
- 热门
- 最新

## MVP TO DO
### 1. 产品目标与边界

- 明确产品定位：内部使用的 AI 官方群聊发现工具，用于行业调研与信息获取，不追求商业化完整度。
- 明确核心价值：减少人工逐个翻官网/GitHub 找官方群入口的时间成本。
- 明确收录边界：**只收录官方群**，不收录民间群、KOL 群、泛社区入口。
- 明确失败策略：宁缺毋滥；搜不到就返回“未发现官方群”。

### 2. 核心使用场景

- 支持主场景：用户输入 **具体产品名**，搜索其官方群。
- 支持输入形式：
    - 产品名
    - 英文别名 / 常见别名
    - 官网域名
- 支持歧义处理：
    - 默认返回前 3 个产品候选
    - 产品名歧义时，返回多个候选供用户选择

### 3. 数据源范围

#### 首批优先搜索源

- 官网首页
- 官网 footer / contact / community 页面
- GitHub 组织主页
- GitHub 仓库 README

#### 后续扩展源

- 官方文档站
- 官方招聘页
- 官方公众号文章页

### 4. 官方群识别规则

- 采用“中等严格”规则：
    - 二维码或入口必须出现在 **官网 / GitHub 官方页面**
    - 周边文案需明显指向：交流、社群、答疑、用户群、开发者群等
- 不展示以下内容：
    - 只有“联系我们”但没有官方群指向
    - 只有公众号二维码但没有明确官方群语义
    - 非官方社区入口

### 5. 平台优先级

#### MVP 优先

- 微信群
- QQ群
- 飞书群

#### 后续再做

- 钉钉群
- Telegram
- Discord

> 其中 **微信群是核心**。

### 6. 搜索结果与卡片设计

#### 结果组织方式

- 一个产品一个主卡片
- 主卡片可展开查看多个官方群

#### 卡片必须字段

- 应用名称
- 入群方式（二维码优先）
- 群类型
- 应用简介
- GitHub stars
- 创建时间
- 最近验证时间

#### 字段口径

- 应用简介：优先使用官网或 GitHub 的一句官方描述
- GitHub stars：优先显示官方主仓库 stars；没有则显示“—”或“无公开仓库”
- 创建时间：默认显示官方主仓库创建时间；没有则显示“未知”
- 最近验证时间：定义为 **系统本次搜索成功抓到该官方群入口的时间**

### 7. 群类型枚举

- 交流群
- 答疑群
- 售后群
- 招募 / 内测群
- 未知

### 8. 展示与交互

- 卡片中展示：
    - 二维码缩略图
    - 放大查看能力
- 若没有二维码但有官方群入口：
    - 展示入口链接
    - 标注“二维码暂未抓取成功”
- 不向用户展示来源链接
- 搜索框位于页面最顶部
- 搜索框下方放筛选栏
- 搜索结果展示在筛选栏下方

### 9. 筛选与排序

#### 筛选项

- 群平台：微信 / QQ / 飞书
- 群类型：交流群 / 答疑群 / 售后群 / 招募/内测群
- 隐藏已添加开关
- 热门
- 最新

#### 默认排序

1. 产品名匹配度
2. 是否有明确官方群二维码
3. GitHub stars

#### 热门定义

- MVP 阶段先只使用 GitHub stars

### 10. 已添加 / 隐藏逻辑

- “已添加”含义：我已经看过了 / 我已经加入了，不想再重复出现
- 默认隐藏已添加结果
- 提供“查看已隐藏结果”开关
- 隐藏粒度：**只隐藏当前群卡片**，不隐藏同产品下的其他群
- 状态保存方式：**当前用户长期保存**
- MVP 不做登录，默认本地使用、本地保存状态

### 11. 更新机制

- 采用 **弱更新**
- 每次搜索时实时抓取最新页面信息
- 重点解决：官方群二维码被替换但旧结果未更新的问题

### 12. 搜索失败处理

- 未发现官方群时：
    - 显示“未发现官方群”
    - 返回空结果
- 不返回“联系我们”“公众号”“疑似社区”等替代项

### 13. MVP 不做的内容

- 不做人工补录
- 不做复杂用户系统 / 登录体系
- 不做团队共享状态
- MVP 不做推荐关键词生成
- MVP 不做复杂热门算法或复杂推荐算法
- 不做大而全的平台覆盖
- 不做泛社区发现，仅做官方群发现

补充说明：
- “推荐关键词”属于后续迭代方向，但当前不进入正式需求范围。
- 后续若实现，优先采用“推荐词入口”形态，而不是让用户直接输入泛关键词全网找二维码。
- 后续初版可先用规则、预置词库和热门源实现，不要求复杂推荐模型。
