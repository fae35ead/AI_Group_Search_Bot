## 目前的搜索策略

![[Pasted image 20260410095251.png]]

若搜索某关键词，设置返回结果为50，可能会发生：
1. GitHub 搜索可能只找到 **8 个候选仓库**
2. 候选扩展补了 **3 个**，Web 兜底又补了 **5 个** → 共 **16 个候选**
3. 逐个爬取这 16 个候选的页面，提取群入口
4. 其中可能只有 **4 个**候选的页面里真的有群聊链接
5. → 最终返回 **4 个结果**，附带空消息提示：`"未在 GitHub/官网相关页面中发现官方群入口"`
6. 不会因为"不够 50"而去凑不相关的结果

## GPT优化前后对比
| 阶段  | “code” 结果数 | “code” 耗时 | “agent” 结果数 | “agent” 耗时 |  时间差 |  提速% | 单条耗时提升% |
| --- | ---------: | --------: | ----------: | ---------: | ---: | ---: | ------: |
| 1   |         10 |     2m00s |          10 |      1m14s | -46s | 38.3 |    38.3 |
| 2   |         15 |     3m15s |          17 |      3m00s | -15s |  7.7 |    18.5 |
| 3   |         21 |     4m10s |          22 |      3m57s | -13s |  5.2 |     9.2 |
| 4   |         30 |     5m10s |          33 |      5m05s |  -5s |  1.6 |    10.7 |

>   说明：
> - **提速%** = (code耗时 − agent耗时) / code耗时 × 100
> - **单条耗时提升%** = (code秒/条 − agent秒/条) / code秒/条 × 100
>   - code 秒/条：12.0 → 13.0 → 11.9 → 10.3
>   - agent 秒/条：7.4 → 10.6 → 10.8 → 9.24
>  
> - agent 在每个阶段都返回了**相同或更多**的结果，且耗时更短
> - 初始阶段（前10条）提速最显著（38%），符合渐进式加载的设计预期；随着结果数增加，两者趋于接近

## 更新后的搜索策略
```mermaid
flowchart TD
    A["用户输入 query / limit / filters / refresh"] --> B["search_with_job"]
    B --> C{"命中缓存且结果数 >= target_limit ?"}
    C -- 是 --> Z["返回缓存结果"]
    C -- 否 --> D["GitHub 搜索主候选"]
    D --> E["构建 crawl candidates"]
    E --> F{"候选不足或需要补充 ?"}
    F -- 是 --> G["Web fallback candidates"]
    F -- 否 --> H["合并候选列表"]
    G --> H

    H --> I["按批次执行 _collect_cards"]
    I --> J["对每个 candidate 执行 _fetch_candidate_pages"]

    J --> K["抓取 seed pages: GitHub repo / homepage"]
    K --> L{"seed page 有 discovery signal ?"}
    L -- 否 --> Y["该 candidate 暂不继续扩展"]
    L -- 是 --> M["extractor.extract(seed pages)"]

    M --> N{"seed pages 已发现官方群入口 ?"}
    N -- 是 --> O["保留 seed 提取结果"]
    N -- 否 --> P["收集 related links<br/>community / qq / wechat / discord / feishu / docs 等"]
    P --> Q["抓取 related pages"]
    Q --> R["extractor.extract(related pages)"]
    R --> S["合并 seed + related 提取结果"]
    O --> S

    S --> T{"仍未在官网主页发现官方群入口，且满足 browser fallback 条件 ?"}
    T -- 是 --> U["Playwright 抓取官网主页"]
    U --> V["提取 background-image / canvas / 动态渲染图片源"]
    V --> W["extractor.extract(browser page)"]
    W --> X["合并 browser 提取结果"]
    T -- 否 --> AA["进入结果归一化"]
    X --> AA

    subgraph Extractor["extractor.extract 页面内提取逻辑"]
        E1["页面预筛：discovery signal / visual signal"]
        E2["GitHub 页面只扫描 README block<br/>官网页面扫描整页"]
        E3["提取非视觉入口：Discord / QQ / Feishu / WeCom / DingTalk 直链"]
        E4["提取 QQ 群号"]
        E5["提取视觉任务：img / 图片型 a 标签 / browser extra visual source"]
        E6["下载图片前规范化 URL<br/>GitHub blob/raw -> raw.githubusercontent.com<br/>camo -> 原图 URL"]
        E7["二维码分析：detect + preprocess + white border + sliding window + hard preprocess"]
        E8["decoded payload 归因"]
        E9["过滤公众号/非群入口<br/>如 weixin.qq.com/r/..."]
        E10["允许自定义短链二维码兜底<br/>前提：已检测到二维码 + 平台/群语义明确 + 非账号语义"]
        E11["生成候选：qrcode / link / qq_number"]
        E1 --> E2 --> E3
        E2 --> E4
        E2 --> E5 --> E6 --> E7 --> E8 --> E9 --> E10 --> E11
    end

    M -.调用.-> Extractor
    R -.调用.-> Extractor
    W -.调用.-> Extractor

    AA --> AB["按 platform + image digest / normalized link / qq number 去重"]
    AB --> AC["过滤仅支持平台：微信 / QQ / 飞书 / Discord / 企业微信 / 钉钉"]
    AC --> AD["构建 ProductCard"]
    AD --> AE{"群入口可落为哪种 entry ?"}
    AE -- qrcode_verified + image_bytes --> AF["保存二维码图片 -> QRCodeEntry"]
    AE -- QQ 群号 --> AG["QQNumberEntry"]
    AE -- 可靠直链 --> AH["LinkEntry"]
    AF --> AI["加入 official_groups"]
    AG --> AI
    AH --> AI

    AI --> AJ["全局去重不同 candidate 的重复群入口"]
    AJ --> AK["生成结果列表"]
    AK --> AL{"需要后台继续补齐更多结果 ?"}
    AL -- 否 --> AM["返回结果 / 写入缓存"]
    AL -- 是 --> AN["创建后台 job 继续抓取剩余 candidates"]
    AN --> AM
```

- 先找候选仓库/官网，再抓 repo_url 和 homepage 两类 seed page；有信号就提取；不够再抓 related page；最后官网主页才可能走 Playwright 动态兜底。
- 返回结果可能是 二维码 / 链接 / QQ群号，最后会去重成卡，不是简单把 GitHub 和官网结果并排拼上。
