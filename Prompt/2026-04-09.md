## 已知问题
- `SearchService.search()` 是单次请求内串起整条搜索流程：查缓存、GitHub 搜索、扩展候选、web fallback、抓页面、提取群入口、合并结果、过滤 viewed，再返回 。  虽然 `_collect_cards()` 内部用了线程池并发抓候选页，但并发只有 `CANDIDATE_FETCH_MAX_WORKERS = 4`，而且浏览器 fallback 预算、相关页上限、web fallback 数量都写死在常量里 。前端这边，返回数量滑条增大后，会触发 `expand` 模式再次请求搜索，逻辑上属于“用户调大结果数 -> 再跑一次后端搜索” 。
	- 改进方向：先做最小拆分：
		1. 搜索阶段和抓取阶段分开
		2. 先返回第一批结果，再异步补更多
		3. 引入 job / task 机制，自动搜索不要直接绑在一次 HTTP 请求里
		4. Playwright 改成复用 browser/context，不要每次临时起
