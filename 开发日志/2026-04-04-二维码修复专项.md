## 2026-04-04 第二轮修复

### 根因分析
GPT 的分析很有价值：
1. **二维码真实性门槛不够硬** - `qrcode_verified` 只是标记，不参与强过滤
2. **平台识别过度依赖 URL/上下文词** - 如果图片 URL 是 `oss.laf.run/xxx.png` 且上下文为空，无法检测
3. **视觉阈值过于硬编码** - 会误放某些截图，也会误杀小而清晰的真实码

### 修复内容

#### 1. 重构 `_extract_from_img_tag()` 分层逻辑

**层级1**：OpenCV decode 成功 → 确认为二维码，qrcode_verified=True

**层级2**：OpenCV detect 成功（有二维码特征）但 decode 失败 + 文本有群意图 → 降级为 link 候选

**层级3**：没有二维码特征，但 URL/上下文有平台关键词 + 有群聊意图 + 尺寸足够 → 降级为 link 候选

**其余**：丢弃

#### 2. 重构 `_extract_from_link_tag()` 图片处理部分

复用相同的分层逻辑

#### 3. 简化 `_should_consider_visual_candidate()`

只做快速预检：必须有平台关键词

## 2026-04-04 第三轮修复：URL 末尾点问题

### 问题
搜索 "FastGPT" 时出现 `SSL: CERTIFICATE_VERIFY_FAILED` 错误，导致请求挂起。

**根因**：Bing RSS 返回的 URL 包含末尾点（如 `https://ai.fastgpt.in.`），Python `urlparse` 保留末尾点，导致：
1. SSL 证书验证失败（证书是发给 `fastgpt.io` 而不是 `ai.fastgpt.in.`）
2. `_domain_key()` 的 `host.split('.')` 产生空字符串

### 修复内容

1. **`_normalize_homepage()`** (`service.py:204-218`)
   - 新增：去除 URL 末尾点

2. **`_domain_key()`** (`service.py:408-416`)
   - 新增：去除 hostname 末尾点

## 技术方案

### P0.2 修复方案

**现状代码分析**：

`entry_extractor.py:_passes_visual_checks()` 当前逻辑：
```python
# 尺寸检查
if width < 180 or height < 180: return False
# 比例检查
if ratio > 1.35: return False
# 弱验证：满足"宽高>=400 且 大小>=20KB" 即通过
return width >= 400 and height >= 400 and len(image_bytes) >= 20_000
```

**问题**：只要图片尺寸够大+文件够大就通过，不验证是否真的是 QR 码

**修复方案**：
1. `_decode_qrcode()` 默认应该被调用（当前被 `enable_opencv_qr_decode` 环境变量关闭）
2. 增加更严格的预筛规则：
   - URL/文件名必须包含平台关键词 OR 上下文必须包含群聊语义
   - 单独尺寸达标不够，必须同时满足"平台/语义"条件
3. OpenCV 解码作为最终验证：能解码出内容 → `qrcode` 类型；解码失败 → 丢弃

**轻量替代方案验证顺序**：
1. URL 包含平台关键词（wechat/qr/feishu/discord）→ 进入候选
2. 上下文（附近文字）包含群聊语义 → 进入候选
3. 尺寸 >= 400x400 且比例接近正方形 → 进入候选
4. OpenCV 解码验证 → 成功 = 真实 QR
