# 2026-04-02 统一探索搜索改造

## 背景
原实现的主语义仍是“给定具体产品，寻找其官方群”，这会导致两个明显问题：

1. 宽泛关键词如 `ChatGPT`、`agent`、`RAG` 的召回率天然偏低。
2. 即使已经发现了相关工具，只要没找到官方群，整张卡片就会消失，前端把“无群”误当成“无结果”。

## 本轮改动
### 1. 产品口径调整
- 将产品重新定义为：`AI 工具发现器 + 群入口发现器`
- 搜索统一走探索式召回
- 无群工具也必须返回卡片

### 2. 后端改动
- `ProductCard` 扩展：
  - `group_discovery_status`
  - `official_site_url`
  - `github_repo_url`
- `groups` 允许为空
- `SearchService` 改为：
  - 官方产品通道
  - GitHub 生态通道
  - 候选去噪、合并、排序
  - 对前 10 个候选逐个做官方群发现
- `empty_message` 只在零候选工具时返回

### 3. 前端改动
- 搜索框文案改为支持：
  - AI 工具名
  - 关键词
  - 官网域名
  - GitHub 仓库
- 结果卡片支持：
  - 已发现官方群
  - 未发现官方群
- 无群卡片展示官网 / GitHub 来源链接

### 4. 文档改动
- `产品总纲.md`
- `PRD.md`
- `ARCHITECTURE.md`
- `产品说明.md`

已统一到新口径，不再按“唯一官方产品卡片”描述。

## 为什么这样改
- GitHub 上大量维护中的软件项目本身就是有效的 AI 工具结果，不应被当成噪音全部过滤。
- 宽泛关键词搜索的目标，本来就不是“唯一官网”，而是“相关工具集合”。
- “没找到官方群”不等于“没找到工具”，前端必须能表达这层差异。

## 验证
- 后端单测：通过
- 后端编译：通过

## 已知问题
- GitHub 生态排序仍是第一版，后续还需要继续调优 repo 去噪规则。
- 前端当前还没有更细粒度地区分“已验证二维码”和“未校验二维码图片”。
- 真实查询 `ChatGPT / agent / Minimax / Claude` 仍需继续做一轮线上样例验收。
