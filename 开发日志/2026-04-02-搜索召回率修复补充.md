# 2026-04-02 搜索召回率修复补充

## 本轮完成

- 已完成“二维码提取稳定化 + 子页发现增强”第一轮后端修复。
- 主方案未变，仍然是“官方网页 HTML / DOM 提取为主，GitHub API / GitHub 搜索为辅”。
- 本轮没有引入付费外部搜索 API，没有放宽到 Discord / Telegram，也没有改动前端功能范围。

## 具体改动

### 1. 候选页发现增强
- `PageFetcher` 新增候选页打分与排序能力，不再只用少量内链直接抓取。
- 站内候选页从单层抓取升级为“两层官方页面发现”：
  - 先抓官网首页 / GitHub README
  - 再抓高分子页
  - 再抓二层高分子页
- 候选页关键词扩展到：
  - 强信号：`community / group / join / invite / wechat / weixin / qq / feishu / lark / qr / qrcode / 社群 / 社区 / 群 / 微信 / 飞书 / 二维码 / 扫码 / 加群 / 入群 / 答疑`
  - 弱信号：`news / blog / event / developer / ecosystem / 活动 / 新闻 / 开发者 / 生态`
- 对 `Minimax` 这类双官方域名产品，已支持把 `minimax.io` 与 `minimaxi.com` 作为同品牌官方根站一起纳入抓取种子。
- 当首页与站内候选页仍无结果时，会对所有已识别官方域名执行轻量 `site:` 兜底搜索。

### 2. 二维码提取稳定化
- `EntryExtractor` 已支持更多图片来源：
  - `src`
  - `data-src`
  - `data-original`
  - `data-lazy-src`
  - `data-canonical-src`
  - `srcset`
- 已支持 GitHub `camo` 图片 URL 解包，以及 Next.js `/_next/image?url=...` 原图 URL 解包。
- 二维码策略已调整为：
  1. 先做 OpenCV 多策略解码
  2. 解码成功则返回已验证二维码
  3. 解码失败但图片像二维码，且位于强官方群上下文中时，保留为“未校验二维码图片”
- `ResultNormalizer` 现已优先保留二维码图片结果，不再让同一上下文的大量链接结果盖掉二维码。

### 3. README 噪音压制
- 已缩小上下文提取范围，不再简单把整块 README 正文往上拼。
- GitHub README 中的目录锚点、章节锚点、`Permalink` 链接、回到顶部等噪音链接会被过滤。
- 链接候选现在必须命中平台特征，不能仅靠宽泛上下文误判为官方群入口。

### 4. Trace 与可观测性
- `SearchTrace / FetchTrace / ExtractionStats` 已扩展，可定位：
  - 抓到了哪些候选页
  - 候选页评分与来源
  - 每页扫描了多少标签
  - 图片候选 / 链接候选数量
  - 图片下载失败数
  - 二维码解码成功数
  - 解码失败但保留为未校验图数
  - 链接噪音过滤数
  - 最终去重数

## 本轮结论

- `FastGPT` 已从“群区命中但二维码抓不出来”提升到“可返回二维码图片结果”。
- `Minimax` 已从“只抓错站 / 只抓首页”提升到“可同时抓多官方域 + 多子页”，但官方群入口仍未命中。
- 当前剩余问题已经从“基础链路失效”收敛到：
  - 少数产品的官方群二维码位置仍未被候选页策略覆盖
  - 部分页面存在二维码图片，但缺乏足够的平台文本线索
  - 真实页面中也可能只有活动报名二维码或非微信 / QQ / 飞书入口
