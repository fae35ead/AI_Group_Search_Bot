# 1. 产品定义
- 产品名称：AI 群聊发现器
- 一句话定义：输入关键词后，先召回相关 AI 工具，再继续发现每个工具的官方群入口。
- 当前主目标：降低"找 AI 工具"和"找对应官方群入口"两件事的人工成本。

# 2. 搜索主语义
- 搜索不再以"唯一官网 / 唯一官方仓库"为唯一目标。
- 搜索统一走探索式召回。
- 宽泛关键词如 `ChatGPT`、`agent`、`RAG`、`MCP`，应返回多个相关工具。
- 精确输入如产品名、官网域名、GitHub repo URL，仍可进入同一条链路，但最相关、最官方的结果要排在前面。

# 3. 候选工具边界
- GitHub 上"维护中的软件项目"可以作为结果本体。
- 不要求候选一定是某个大品牌的上游官方产品。
- 仍要过滤或显著降权以下仓库：
  - `awesome` 列表
  - 教程 / guide / tutorial
  - 纯 SDK
  - prompt 集合
  - 模板 / boilerplate / starter
  - fork / archived / disabled

# 4. 群入口边界
- 只收录微信 / QQ / 飞书 / Discord 官方群入口。
- 对品牌型产品：只认其官网 / 官方 GitHub。
- 对 repo 型工具：该 repo README、项目主页、项目官网就是它自己的官方源。
- 仍然不从第三方帖子、导航站、论坛回帖或泛社区页面收录群入口。

# 5. 结果规则
- 单次搜索默认返回最多 10 个工具卡片。
- 有群的工具：展示官方群入口。
- 无群的工具：本轮不展示结果卡片。
- `empty_message` 在"没有召回到相关工具"或"召回到工具但未发现受支持官方群入口"时出现。

# 6. 当前搜索输入
- AI 工具名
- 产品名
- 宽泛关键词
- 官网域名
- GitHub repo URL

# 7. 后端固定流程
1. 标准化输入。
2. 候选工具召回。
3. 候选工具去噪、合并、排序。
4. 对前 10 个候选工具逐个执行官方群发现。
5. 只返回已发现官方群入口的工具卡片。

# 8. 候选召回来源
- 官方产品通道：用户输入精确产品名时，通过 Bing RSS 发现官网站点，结合 GitHub API homepage 字段，优先召回官方产品。
- GitHub 生态通道：按关键词搜索 repo，召回多个相关软件项目。
- 关键词推荐通道：用户打开页面时，后端返回 GitHub topic:ai 热门仓库排名（前 12），作为探索入口；点击推荐词后复用主流搜索链路。
> 说明：群入口发现范围覆盖 GitHub 页面 + 官网主页 + 官网相关内链（community/support/join 等），以 GitHub 仓库为主链路，官网发现为补充。

# 9. 排序原则
1. 精确产品词 / 域名 / repo URL 命中优先。
2. repo name / description / topics / README 命中优先。
3. homepage 存在优先。
4. stars 和近期活跃度参与排序。

> 补充：前端支持对已有结果进行二次筛选（星级阈值、创建时间范围），筛选为客户端即时过滤，不重新请求后端。

# 10. API 输出口径
- `ProductCard.groups` 当前仅在已发现官方群时返回，列表内至少包含 1 条入口。
- `group_discovery_status` 当前仍保留字段，但结果卡片默认应为 `found`。
- 卡片补充字段：
  - `official_site_url`
  - `github_repo_url`
- `GET /api/recommendations`（关键词推荐接口）：
  - 返回 GitHub topic:ai 热门仓库排名前 12（经去噪过滤）
  - 带 30 分钟 TTL 内存缓存
  - 响应格式：`{tools: [{name, fullName, stars, description, topics}], cached_at}`

# 11. 前端展示口径
- 首页文案要明确支持关键词、官网域名和 GitHub 仓库。
- 首页文案要明确支持微信 / QQ / 飞书 / Discord。
- 结果列表只展示已发现官方群的工具卡片。
- "没找到官方群"应统一进入空结果提示，而不是展示空群卡片。

# 12. MVP 不做
- 不接入付费外部搜索 API。
- 不改成全网搜二维码。
- 不收录 Telegram / 钉钉（微信 / QQ / 飞书 / Discord 均在 MVP 支持范围内）。
- 不做用户侧"模式切换"。

# 13. Agent 约束
- 所有 agent 必须以本文为唯一口径。
- 任何文档、代码和 UI 都不能再把产品定义回退成"只适合具体产品名的官方群发现器"。
- Reviewer 必须重点检查：
  - 是否只返回已发现官方群的工具卡片
  - 是否仍只从候选工具自己的官方页面提取群入口
  - 是否把 GitHub 生态项目当成有效结果本体

# 14. 版本记录
- `2.0`：统一搜索改造为"AI 工具发现 + 群入口发现"。
- `2.1`：新增关键词推荐通道（/api/recommendations），明确群入口发现范围覆盖 GitHub + 官网 + 官网内链，前端支持客户端筛选。
