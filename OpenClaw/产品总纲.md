# 1. 产品是什么

**产品名称**：AI群聊发现器
**一句话定义**：帮助用户快速找到 AI 产品官方群聊（微信/QQ/飞书）入口的内部调研工具。

**核心价值**：减少人工逐个翻官网/GitHub 寻找官方群入口的时间成本。

**产品定位**：
- 内部使用，不追求商业化完整度
- 收录范围：只收录**官方群**，不收录民间群、KOL 群、泛社区入口
- 失败策略：宁缺毋滥；搜不到就显示"未发现官方群"

**背景**：
为了获取 AI 行业前沿知识并提升调研效率，我们需要快速发现并加入优秀、领先、热门 AI 应用的官方群聊。这些官方群聊是重要信息源，但目前只能手动在各个官网/GitHub 逐一查找加入，非常繁琐。本工具旨在自动化这一发现过程。

# 2. 术语表（Agent 通用语言）

| 术语         | 定义                                                                        | 注意事项                  |
| ---------- | ------------------------------------------------------------------------- | --------------------- |
| **官方群**    | 在**官网首页、footer/contact/community**或 **GitHub 官方页面**（非民间仓库）上展示的、由官方运营的群聊入口 | 必须与"联系我们"区分，后者不展示     |
| **入群方式**   | 用户加入群的方式，以**二维码**为主，无二维码时展示入口链接                                           | 二维码抓取失败时标注"二维码暂未抓取成功" |
| **最近验证时间** | 系统本次搜索**成功抓取**到该官方群入口的时间                                                  | 用于判断二维码是否可能已过期        |
| **已添加**    | 用户标记"我看过了"或"我已加入"的状态                                                      | 目的：过滤重复推荐，无需拆分为两个状态   |
| **弱更新**    | 用户每次搜索时，实时抓取最新网页信息                                                        | 每次都重新抓，不依赖缓存          |
| **热门**     | 以 GitHub stars 为量化指标（MVP 阶段唯一指标）                                          | 后续可扩展                 |
| **最新**     | 按最近验证时间倒序                                                                 | MVP 功能之一              |
| **候选产品**   | 搜索时因产品名歧义返回多个匹配项，供用户自行选择                                                  | 默认返回前 3 个候选           |
| **推荐关键词** | 面向探索场景的后续能力，用于生成可点击主题词；点击后先映射到候选产品，再复用官方群发现流程                        | 不属于当前 MVP 默认输入，不等于全网搜二维码 |
| **主卡片**    | 每个产品对应一个主卡片，可展开查看多个群                                                      | 展开前显示应用名+概览           |

# 5. 数据源范围

### 首批优先搜索源（MVP）
1. 官网首页
2. 官网 footer / contact / community 页面
3. GitHub 组织主页
4. GitHub 仓库 README

### 后续扩展源
- 官方文档站
- 官方招聘页
- 官方公众号文章页

### 搜索输入支持形式
- 产品名（中文/英文）
- 英文别名 / 常见别名
- 官网域名

### 正式搜索触发规则
- 正式搜索只在用户**按 Enter**、**点击搜索按钮**或**点击候选产品**后发起。
- 输入过程可用于更新候选产品提示，但**不得**在用户仅输入时直接触发正式抓取。

### 后续扩展：推荐关键词入口（当前不启用）
- 推荐关键词属于后续迭代能力，不改变当前搜索输入定义。
- 推荐关键词用于探索模式：用户点击推荐词后，系统先扩展候选产品，再复用既有官方群发现流程。
- 推荐关键词不是“直接全网搜二维码”，结果仍只来自官网 / GitHub 官方页面。
- 推荐词示例：`热门 AI 编程助手`、`开源 AI Agent`、`AI 视频生成`、`最近新增 AI IDE`

### 搜索失败处理
- 明确提示"未发现官方群"，返回空结果
- **不**展示：只有"联系我们"但没有群指向、只有公众号二维码、非官方社区入口

# 6. 卡片字段定义（必须严格遵循）

### 主卡片字段
| 字段名 | 来源/计算方式 | 空值处理 |
|--------|---------------|----------|
| 应用名称 | 用户搜索词匹配 | — |
| 入群方式 | 抓取二维码 / 入口链接 | 二维码抓取失败→展示链接并标注"二维码暂未抓取成功" |
| 群类型 | 识别周边文案语义（交流/答疑/售后/招募/未知） | 显示"未知" |
| 应用简介 | 优先用官网或 GitHub 的一句官方描述 | 显示"—" |
| GitHub stars | 官方主仓库 stars | 显示"—"或"无公开仓库" |
| 创建时间 | 官方主仓库创建时间 | 显示"未知" |
| 最近验证时间 | 本次搜索成功抓取的时间 | — |

### 字段规则
- **不展示**来源链接（给用户）
- 卡片可展开，同一产品下展示多个群
- 搜索结果数量上限：前 3 个候选产品

# 7. 展示与交互规范

### 页面结构
```
┌─────────────────────────────────┐
│           搜索框（最顶部）         │
├─────────────────────────────────┤
│ 筛选栏（平台/群类型/热门/最新/隐藏开关）│
├─────────────────────────────────┤
│         结果区（主卡片列表）       │
└─────────────────────────────────┘
```

### 二维码展示
- 缩略图 + 点击放大查看
- 二维码优先；没有二维码时展示入口链接

### 空结果态
- 提示"未发现官方群"，不展示任何替代项

### 已隐藏结果态
- 默认隐藏"已添加"状态
- 提供"查看已隐藏结果"开关（切换）

### 隐藏逻辑
- 隐藏粒度：**只隐藏当前群卡片**，同产品其他群仍显示
- 状态保存：当前用户本地长期保存（MVP 无需登录）

# 9. OpenClaw Agent 架构与分工

```
┌─────────────────────────────────────────────────────┐
│                    主 Agent（Planner）                 │
│  - 持有唯一真相版本（产品总纲、开发日志）                │
│  - 维护产品口径，输出统一术语表                          │
│  - 派发任务给子 Agent，合并最终输出                     │
│  - 负责任务规划与进度追踪                               │
└────────────────┬────────────────────────────────────┘
                 │
    ┌────────────┼────────────┬───────────┬──────────┐
    ▼            ▼            ▼           ▼          ▼
  Frontend   Backend      QA       Reviewer    (文档 Agent)
```

### 主 Agent（Planner）
- 持有产品总纲和开发日志唯一版本
- 维护统一口径和术语表
- 给每个子 Agent 派发明确任务（含任务编号和 Phase 归属）
- 合并子 Agent 输出，统一风格
- 负责任务进度追踪（更新开发日志/日志.md 的 TODO LIST）
- **禁止**子 Agent 自行修改产品总纲或开发日志

### Frontend Agent
- 负责 Phase 3 前端核心组件开发（见开发日志 § 二）
- 按 DESIGN.md 组件清单实现 UI，不得引入产品总纲未定义的字段
- 与 Backend Agent 对齐 API 接口格式
- 提交前需 QA Agent 通过测试
- **禁止**自行决定产品需求，所有需求必须来自 PRD

### Backend Agent
- 负责 Phase 2 后端核心模块开发（见开发日志 § 二）
- 实现搜索流程 8 步（见本文档第11节），不得增删步骤
- 实现 FastAPI 接口与 SQLite 本地存储
- **禁止**引入登录态、远程数据库、付费 API
- **禁止**把推荐关键词并入当前 8 步主流程

### QA Agent
- 负责 Phase 4 集成测试与稳定化（见开发日志 § 二）
- 执行端到端测试，覆盖主要用户路径
- 记录已知问题到开发日志/日志.md § 四
- 测试通过后提交 Reviewer Agent 做最终把关

### Reviewer Agent
**角色定位**：代码审查、架构检查、发布前把关，是进入”可合并/可发布”前的最后一道检查。

#### 三大职责

**1. 代码审查**
- 逻辑正确性
- 边界条件处理
- 错误路径覆盖
- 是否存在明显回归风险
- 是否引入与需求无关的改动

**2. 架构检查**
- 模块职责是否清晰
- 前后端边界是否合理
- 是否出现过度设计
- 是否出现耦合过深或难以维护的实现

**3. 发布前把关**
- 必改问题（🔴 阻断）
- 建议优化（🟡 建议）
- 风险项
- 可发布 / 不建议发布 结论

#### 约束
- 不负责重新发明需求
- 不负责替代 Frontend 或 Backend 重新实现功能
- 不把风格偏好当成高优先级缺陷
- 必须优先报告真实风险和行为问题

#### 输出格式

```
## Reviewer 输出

### 必改问题（🔴）
- <问题>

### 建议优化（🟡）
- <建议>

### 风险项
- <风险>

### 缺失项
- <缺失>

### 结论
- 可发布 / 不建议发布
```

#### 审查检查清单（10 项）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 术语一致性 | 所有文档使用产品总纲术语表中的标准词 |
| 2 | 字段覆盖 | PRD → ARCH → DESIGN → Frontend 实现 四层覆盖链完整 |
| 3 | 平台列表 | 仅 MVP 三平台（微信/QQ/飞书），无多余平台 |
| 4 | 群类型枚举 | 5种枚举严格一致：交流群/答疑群/售后群/招募-内测群/未知 |
| 5 | 搜索流程 | Backend 的 8 步流程与本文档第11节完全一致 |
| 6 | 空值处理 | 所有字段的空值处理方式在文档与代码中一致 |
| 7 | MVP 边界 | 所有文档与代码均无”不做清单”之外的内容 |
| 8 | 字段数量 | DESIGN 与 Frontend 实现的主卡片字段严格等于 7 个 |
| 9 | 搜索输入口径 | 当前输入始终为产品名/别名/域名，推荐关键词明确标注为后续入口 |
| 10 | 官方源边界 | 推荐关键词相关描述坚持只认官网/GitHub 官方页面 |

# 11. 搜索流程（8步，数据流规范）

> Agent ARCHITECTURE 必须严格遵循此流程，不得自行增删步骤。

> 说明：后续若增加“推荐关键词”能力，应作为 Step 1 之前的扩展入口，不并入当前 8 步主流程。

```
Step 1: 搜索流程
         用户输入产品名/别名/域名
              ▼
Step 2: 页面抓取流程
         并行抓取：官网首页 + footer + community + GitHub org + README
              ▼
Step 3: 官方源识别
         过滤非官方页面，保留官方域名下的页面
              ▼
Step 4: 二维码/链接提取
         定位二维码图片 OR 入群链接
              ▼
Step 5: 群类型识别
         分析周边文案：交流/答疑/售后/招募/未知
              ▼
Step 6: 结果归一化
         填充卡片字段（应用名/群类型/简介/stars/时间等）
              ▼
Step 7: 本地状态存储
         对比"已添加"列表，过滤已隐藏群
              ▼
Step 8: 弱更新机制
         记录本次验证时间，返回最新结果
```

# 12. MVP 边界清单（Agent 不得越界）

以下内容 MVP **不做**，子 Agent 不得引入相关设计：

| 不做项 | 说明 |
|--------|------|
| 人工补录 | 我们追求自动化搜索，不做手动补充 |
| 登录体系 | 无需登录，本地使用，本地保存状态 |
| 团队共享状态 | 单用户本地存储 |
| 推荐关键词生成 | 后续迭代能力，当前 MVP 不做 |
| 复杂热门算法 / 复杂推荐算法 | MVP 仅用 GitHub stars；后续若做推荐关键词，初版也应优先规则化实现 |
| 全部平台覆盖 | MVP 仅微信/QQ/飞书 |
| 泛社区发现 | 只做官方群发现 |
| 非官方群 | 不展示民间群、KOL群 |
| 替代入口 | 不展示只有"联系我们"、只有公众号二维码的页面 |
